# classroom-app UI設計基準書 v2 — iPad 11インチ(M4)横向き最適化

Claude Code への実装指示書。目標: スクロール最小化・タッチミス防止・視覚的明快さ。
開発ルール厳守: str_replace差分編集のみ / CACHE_VERSION up / node --check / push は承認後。

## 前提となる物理制約

- iPad 11インチ横向きの論理解像度: **1180 × 820pt**(ステータスバー等を引いた実用高さ 約770pt)
- サイドバー60px を引いた コンテンツ幅: **約1120pt**
- 指先の接触面: 平均7〜10mm ≒ 44pt — **Apple HIG の最小タッチターゲット 44×44pt**
- Smart Keyboard Folio 使用時は画面が傾斜し腕を伸ばしてタップ → 画面上端ほどミスが増える。**頻出操作は画面の中央〜下部・右側に置く**(右手操作前提)

## 基準1: タッチターゲット 44pt ルール

現状の実測: min-height 28〜34px のボタンが主流(タッチミスの主因)。

- すべてのタップ対象は **44×44px 以上**。例外なし
- 見た目を大きくしたくない場合は `padding` または擬似要素でヒット領域のみ拡張:
  ```css
  .small-btn { position: relative; }
  .small-btn::after { content:''; position:absolute; inset:-8px; }
  ```
- 隣接するタップ対象の間隔は **8px 以上**(誤タップ防止)
- 削除など破壊的操作のボタンは、他のボタンから **16px 以上** 離す か 長押し/確認ダイアログ
- 実装方法: CSS変数 `--tap-min: 44px;` を定義し、共通クラス `.tappable { min-height: var(--tap-min); min-width: var(--tap-min); }` を順次適用。新規UIは必ずこのクラスを使う

## 基準2: タイプスケール4段階

現状: 0.65〜0.95rem に10種類以上が混在。

| 役割 | サイズ | 用途 |
|------|--------|------|
| `--fs-caption` | 12px | 補足・ラベル・単位 |
| `--fs-body` | 14px | 本文・ボタン・表のセル |
| `--fs-title` | 17px | 見出し・現在選択中の児童名 |
| `--fs-display` | 22px | 画面タイトル・重要な数値 |

- **0.65rem(約10.4px)は全廃**。11px未満は iPad では実用的に読めない
- フォントウェイトは **400 と 700 の2種のみ**(600や混在をやめる)
- 移行方法: `:root` に変数を定義 → 画面単位で `font-size:0.xx rem` を変数に置換。一括置換はせず画面ごとにレビュー

## 基準3: 連続入力モード(タップ数半減)

最頻出作業 = 全児童への状態入力(提出物・忘れ物・成績・出席)。

現状フロー: 「児童を選ぶ → 状態を押す」 = 児童数 × 2 タップ
新フロー: 「状態を押すだけで自動的に次の未入力児童へ」 = 児童数 × 1 タップ

**仕様:**
1. 画面右側に固定の状態パレット(52px以上の大ボタン、最頻出の状態を最上段)
2. 「入力中の児童」を大きく表示(17px以上)+ グリッド上で枠ハイライト
3. 状態ボタンを押す → 現在の児童に記録 → **次の未入力児童へ自動前進**
4. グリッド上の任意の児童をタップすればそこへジャンプ(割り込み修正)
5. 進捗カウンタ「24 / 35」を常時表示
6. 全員入力完了時にトースト + 完了表示

**適用画面(優先順):** ①提出物チェック ②忘れ物チェック ③成績入力(recモジュール — 数値入力はテンキーパレット化) ④出席簿

## 基準4: 1画面フィット(スクロールゼロ)

- 児童グリッドは **7列**を基本に全員を1画面表示(38人 → 7×6 で収まる)
- セルの最小: 幅 130pt × 高さ 56pt(44ptルール+名前の可読性を満たす)
- 検算: 7列 × 130pt = 910pt < 1120pt ✓ / 6行 × 56pt + 隙間 = 約 380pt、ヘッダー・パレット込みでも 770pt 内 ✓
- リスト表示が必要な画面(課題一覧など)は「直近をN件表示 + もっと見る」でファーストビューにスクロールを出さない
- 縦向き(820×1180)は対象外とする(横向き専用と割り切り、既存の運用と一致)

## 基準5: 色の意味を固定する

状態色をアプリ全体で統一(画面ごとに違う色を使わない):

- 提出/出席/合格 = 緑系
- 遅れ/要注意 = 橙系
- 未提出/欠席/不合格 = 赤系
- 未入力 = 白+グレー枠
- 選択中 = 紫枠(2px)

色だけに頼らず、必ず **記号(✓/遅/未/欠)を併記**(色覚多様性対応。教室の投影時にも判別可能)。

## 実装の進め方(推奨順)

1. `:root` にCSS変数(`--tap-min`, `--fs-*`)と状態色を定義 — 1コミット
2. 提出物チェック画面に連続入力モードを実装(基準3+4の実証) — 1コミット、実機で1週間試用
3. 試用フィードバックを反映後、忘れ物チェック → 成績入力へ横展開
4. タイプスケール統一は画面を触るたびに「ついで適用」(専用の大規模リファクタはしない)

## やらないこと

- 全画面同時の大規模リファクタ(リスクが利益を上回る)
- アニメーション・装飾の追加(目的は速度と確実さ)
- 縦向き対応・スマホ対応(iPhone はビューワー用途と割り切る)
