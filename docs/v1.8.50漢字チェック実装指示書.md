# v1.8.50 実装指示書：漢字チェックの二層構造化

このファイル単体で完結した指示書です。他の資料を参照する必要はありません。

## 0. 役割と開発ルール（厳守）

- このセッションの役割：classroom-app（単一ファイルSPA、index.html）の実装。
- **データ保存形式は一切変更しない。** 既存の保存キー・データ構造を変えてはならない。
- **入力方式（循環：空→△→○）は変更しない。** 今回変えるのは画面構造と動線のみ。
- 編集は **str_replace 差分編集のみ**。create_file で大きなファイルを書き直さない。
- push 時は **sw.js の CACHE_VERSION を上げる**（今回は v1.8.50）。
- push 前に **node --check で構文確認**。
- **push は y.y の承認後のみ。** 勝手に push しない。
- 本指示書を `docs/v1.8.50漢字チェック実装指示書.md` として保存してから着手すること。

## 1. 背景（なぜこれをやるか）

漢字チェックは「漢字ノートに児童が練習してきたかどうか」を教員がチェックする機能。
現場の2つの痛みを解消するための改修：

- **痛み①：タップミスが多い** → タップ対象が小さいことが主因。44pt ルールを適用して解消する。
- **痛み②：最初の文字から最後の文字まで画面の切り替えが手間** → 1文字ごとに画面を送る現構造が原因。児童単位の二層構造に再編して解消する。

提出物チェック（v1.8.49）で確立した設計思想を漢字チェックへ横展開する第1号。
**ただし入力方式は変えない**——提出物はモードレス化したが、漢字チェックは循環方式（空→△→○）を維持する（y.y の運用上、循環で問題ない／空はそもそもタップしない）。

---

## 2. 設計の土台となる物理制約（iPad 11インチ M4・横向き専用）

- 論理解像度 1180 × 820pt、実用高さ約 770pt、サイドバーを引いたコンテンツ幅約 1120pt。
- 最小タッチターゲットは **44 × 44pt**（Apple HIG）。例外なし。
- 縦向き・スマホは対象外（横向き専用と割り切る）。

---

## 3. 確定仕様

### 3-1. 入力層（児童一人のページ）

- 一覧で児童を選ぶと、その児童（例：Aさん）専用のページが開く。
- **レイアウト：横8列 × 回数ぶんの行。**
  - 1行 = 1回ぶんの漢字（8文字が横に並ぶ）。
  - 回を追うごとに行が下に積み重なる構造。
  - **最新回が最上段**に来ること（古い回ほど下）。ページを開いた瞬間、最上段に最新回が見える状態にする。
  - 8文字は1画面（iPad横）に収まる。回が増えれば縦スクロールでさかのぼれる。
- **各セルの入力方式：循環（空 → △ → ○ → 空 …）を維持。** タップするたびに循環する現行ロジックをそのまま使う。空はタップしない運用。
- **各セルのタップ領域は最低 44pt（`var(--tap-min)`）。** セル間の間隔は 8px 以上を確保し、隣接誤タップを防ぐ。これが痛み①への対策。
- 入力完了後の動線：**「一覧へ戻る」を主動線**とする。「次の児童へ進む」は設けない（ノートが出席番号順に揃わず、回収できた児童から一覧で選ぶ運用のため）。

### 3-2. 一覧層（クラス全体）

- クラス全員の状況を俯瞰できる。現行の一覧ページに近い役割。
- 各児童の行に **「未チェック数（＝まだ ○ も △ も付いていない漢字の累計数）」** を表示する。
  - 範囲は **全回の累計**（最新回だけでなく、これまでの全漢字のうち未チェックの総数）。
  - できていない児童・溜まっている児童が一目で分かること。
- 児童をタップすると、その児童の入力層（3-1）へ遷移する。

### 3-3. 動線まとめ

一覧層（クラス全体・未チェック数の累計表示） → 児童をタップ → 入力層（その児童の 横8列×回数行、最新回が上、循環入力、44ptセル） → 「一覧へ戻る」 → 一覧層

---

## 4. 色・記号（全体で統一）

- 合格・できた = 緑系、要注意 = 橙系、未チェック = 白＋グレー枠。
- 色だけに頼らず必ず記号（○ / △ など）を併記する（色覚多様性・教室投影対応）。
- 既存の漢字チェックの記号（○ / △）と配色を尊重しつつ、CSS変数（`--st-*`）が定義済みであれば流用する。新規で勝手な配色を作らない。

---

## 5. Step 0：事前調査（実装前に報告し、承認を待つこと）

実装に入る前に、漢字チェック画面の現行実装を調査して報告すること。**報告後はいったん停止し、レビュー承認を待つ。**

1. **データ保存構造**：児童 × 漢字 × 回 が、どの保存キーに・どんな形式で保存されているか。「回」の概念がデータ上どう表現されているか（回ごとに区切られているか、フラットな1配列か等）。
2. **循環ロジック**：現行の循環（空→△→○）の実装箇所と、循環結果が保存値へ反映されるパス。
3. **現在の画面描画**：現行は「縦長テーブル、列 = 漢字」と把握している。その詳細と、「回」が画面上どう扱われているかを確認。
4. **児童選択・画面遷移**：児童ごとのページや画面遷移の既存の仕組みがあるか。
5. **【最重要】新構造がデータ形式無変更で実現可能かの判定**：
   - 「横8列 × 回数行・最新回先頭・累計未チェック数」が、**既存データのまま描画変更だけで**実現できるかを判定して報告する。
   - もし既存データに「回」の構造が無く（例：全漢字がフラットに保存されている等）、新構造の実現にデータ形式の変更が必要になる場合は、**実装に進まず、実装案を提示して指示を待つこと。** データ形式無変更が大原則。

---

## 6. 機械検証（push 承認の前提）

実装後、以下をブラウザ E2E（puppeteer-core + ローカル Chrome.app + python3 http.server）で機械検証する。テストハーネスは前回作成済みの `tests/` を流用・拡張し、リポジトリにコミットする（node_modules は .gitignore 済み）。

1. **データ形式バイト一致**：新構造の入力層で循環操作（空→△→○→空）した後の保存値が、現行画面で同一操作をした保存値と **バイト一致** すること（入力方式無変更の裏取り）。
2. **累計未チェック数**：一覧の表示数が、その児童の全回・全漢字のうち未チェック（空）の数と一致すること。
3. **最新回先頭**：入力層を開いた時、最上段に最新回が表示されていること。
4. **44pt**：各セルの実寸が 44pt 以上であること（レンダリング後に計測）。
5. **動線**：入力層から一覧へ戻れること／一覧から児童選択で入力層へ遷移できること。
6. **全体フロー**：一覧→児童選択→循環入力→一覧へ戻る、の一連がエラーなく動くこと。1画面フィット（iPad横想定）。コンソールエラーが無いこと（favicon 404 は除外可）。

---

## 7. 進め方とコミット分割

- Step 0 報告 → レビュー承認 → 実装 → 機械検証 → コミット作成（push 保留）→ 完了報告、の順。
- コミット分割は Step 0 で構造が見えた段階で提案すること。目安：①描画構造の再編（入力層の横8列×回数行・最新回先頭）／②一覧層の累計未チェック数表示／③テストハーネスの拡張。各コミットは独立してレビュー・revert できる粒度に。
- 完了報告では、初回検証で FAIL があった場合「テスト側の修正で解消したか／アプリ側の修正を要したか」を必ず明示すること。

---

## 8. やらないこと

- 入力方式（循環）の変更。
- データ保存形式の変更（必要が生じたら必ず停止して相談）。
- 全画面同時の大規模リファクタ。
- アニメーション・装飾の追加。
- 縦向き・スマホ対応。
