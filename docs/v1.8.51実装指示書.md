# v1.8.51 実装指示書：漢字チェックの単元概念廃止＋データ移行＋児童選択タイル化

このファイル単体で完結した指示書です。他の資料を参照する必要はありません。

---

## 0. 役割と開発ルール（厳守）

- classroom-app（単一ファイルSPA、index.html）の実装。
- 編集は **str_replace 差分編集のみ**。create_file で大ファイルを書き直さない。
- push 時は **sw.js の CACHE_VERSION を上げる**（今回 v1.8.51）。
- push 前に **node --check** で構文確認。**push は y.y の承認後のみ。** 勝手に push しない。
- 本指示書を `docs/v1.8.51実装指示書.md` として保存してから着手すること。
- **【今回の最重要原則】データ移行を伴う。過去の漢字チェック記録（児童につけてきた ○/△）を絶対に失わないこと。** 移行は「読む→変換→検証→（通れば）書く」の順で、検証に失敗したら書かない。

---

## 1. 背景

v1.8.50 で漢字チェックを二層化（一覧層＋児童別8列グリッド入力層）した。今回はその続きで3点：

- **課題1：単元概念の廃止。** 現状データは `units` 配列に分かれているが、ユーザーは「単元」を意識せず「編集で文字を追加するだけ」にしたい。**データ構造も1つにまとめる（フラット化）。これはデータ形式の変更を伴うため、移行を最重要・最慎重に扱う。**
- **課題2：児童選択（一覧層）を出席簿と同じタイルグリッドにする。**
- **課題3：各タイルに未チェック数だけでなく再提出（△）数も表示する。**

前提：実データに複数単元が存在する。移行対象データは小さくない。y.y は事前に一括印刷PDFで全データのバックアップを取得済み（人間側の最終保険）。

---

## 2. データ移行（課題1）— 最重要。独立した最初のコミットとする

**現行形式**（Step 0 で最終確認）：
`{ units: [ {name, chars:[...], checks:{ stuIdx: { charIdx: 'o'|'d' } } }, ... ] }`

**新形式**（具体形は Step 0 報告を受けて確定）：
`{ chars: [全漢字をフラットに連結], checks: { stuIdx: { charIdx: 'o'|'d' } } }` 等のフラット構造。

移行は以下の条件を **すべて** 満たすこと：

- **条件A（検証してから書く）**：旧データを読む → 新形式に変換 → **変換前後で全児童 × 全漢字のチェック値（○/△/未）が1個も変化していないことをコード内で照合** → 照合が通って初めて新形式を保存。**照合に失敗したら新形式を書かず、旧データを保持したまま中断し、コンソールにエラーを出す。**
- **条件B（バックアップ）**：移行前に旧形式データを別キー `spa_kanji_backup_v1851` へ退避してから新形式を書く。退避が成功しなければ移行しない。
- **条件C（冪等性）**：移行処理が二重に走っても壊れないこと。新形式データ（units 無し）に対して移行処理が再度当たっても、データを二重化・破損させない。判定は「`units` キーが存在するか」等の明確なフラグで行う。SW 更新・リロードで初期化が複数回走る前提で設計する。
- **条件D（並び順の保存）**：フラット化で単元境界が消えるので、漢字の並び順を明確に定義する。v1.8.50 の入力層表示順（Step 0 で確認）と移行後の順序を一致させ、ユーザーの体験（最新の漢字がどこに来るか）が変わらないようにする。

## 3. 単元UIの撤去（課題1の続き）

- 単元セレクト・「＋単元追加」「単元削除」など、単元に関する UI 要素をすべて撤去。
- 文字の追加は「編集で文字を足す」だけの操作にする（既存の単元追加モーダルを、単元名なしの「漢字を追加」UI に作り替えるか、シンプルな追加入力にする）。追加文字は新形式 `chars` の末尾／先頭（課題1-Dの表示順に合わせる）に足す。
- `kjBuildFlatItems` など units 前提だった関数は新形式に合わせて簡素化（units→flat の変換層が不要になるはず）。マッピングは「グリッド位置 ↔ charIdx」の単純対応になる。

---

## 4. 児童選択のタイル化（課題2）

- 現状の一覧層（縦テーブル）を、**出席簿と同じタイルグリッド**に変更。出席簿のタイル構造・CSS を流用し、共通操作体系に沿わせる。
- 出席番号順。タップでその児童の入力層へ遷移（既存の `kanjiOpenStudent` を使用）。
- 44pt ルール・タイル間 8px 以上を踏襲。

---

## 5. タイル内の表示（課題3）

- 各タイルに **出席番号 ＋ 児童名 ＋ 未チェック数 ＋ 再提出（△）数** を表示。
- 数字併記を基本とし、**△が1件以上ある児童はタイルに橙系のアクセント（`--st-warn-*`）** を付けて目立たせる。「未チェックのみ」「△あり」が一目で区別できること。
- カウントは新形式の全 `chars` を対象に、未＝checks に値が無い／△＝値が `'d'`、で算出。

---

## 6. Step 0：事前調査（実装前に報告し、承認を待つこと）

1. **現在の実データの規模**：何単元・合計何文字・何児童ぶんのチェックがあるか（移行リスクの大きさの把握）。
2. 現行 `units` 構造の詳細と、v1.8.50 入力層の **現在の表示順**（新しい単元が上か下か）。移行後もこの順を保つため。
3. 単元 追加／削除／編集 の既存 UI・関数の洗い出し（撤去対象の特定）。
4. 出席簿のタイルグリッドの実装（流用元の特定）。
5. **新形式の具体構造の提案** と、移行・冪等性判定の設計案。**この提案へのレビュー承認を得てから移行コードを書くこと。**

---

## 7. 機械検証（push 承認の前提）

1. **【最重要】移行の無損失検証**：現行 units データ（複数単元・複数児童のチェックを含むテストデータ）を投入 → 移行実行 → **移行後の新形式から読めるチェック値が、移行前の全児童 × 全漢字で完全一致** することを照合。1件でも差異があれば FAIL。
2. **冪等性検証**：移行を2回連続実行 → 2回目で何も壊れない（データが1回目と同一）こと。
3. **バックアップ検証**：移行後 `spa_kanji_backup_v1851` に旧データが保持されていること。
4. **データ追加**：文字を追加 → 新形式 `chars` に正しく足され、入力層に反映されること。
5. **循環入力**：空→△→○→空 が新形式 `checks` に正しく保存されること（バイト一致）。
6. **タイル表示**：未チェック数・△数が実データと一致。△ありタイルのアクセント表示。
7. **動線**：タイルタップ → 入力層 → 一覧へ戻る。
8. **一括印刷**（printAllKanjiPDF）が新形式データで動作すること（データ形式変更の影響を再確認）。
9. **全体フロー** ＋ 1画面フィット ＋ コンソールエラーなし（favicon 404 除く）。

---

## 8. 進め方とコミット分割

- Step 0 報告 → **新形式構造と移行設計のレビュー承認** → 実装 → 機械検証 → コミット作成（push 保留）→ 完了報告。
- コミット分割案：①データ移行ロジック（独立・最優先・無損失検証付き）／②単元UI撤去とフラット化対応／③児童選択タイル化＋△表示／④テストハーネス拡張。①は単独で検証・revert できる粒度に。
- 初回検証で FAIL があれば「テスト側の修正か／アプリ側の修正か」を完了報告で明示。

---

## 9. やらないこと

- 移行の無損失検証を省略すること（絶対に省略しない）。
- バックアップなしの移行。
- 単元概念の部分的残存（UIだけ消してデータに残す等。今回はデータも完全フラット化する）。
- アニメーション・装飾の追加、縦向き・スマホ対応。
