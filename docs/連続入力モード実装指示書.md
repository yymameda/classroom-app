## コミット2: 連続入力モードの仕様

まず既存の提出物チェックUI(SUBMISSIONSモジュール内のチェック画面)の構造を調査し、**既存のデータ保存関数(状態の記録ロジック)はそのまま使う**。変えるのは入力UIのみ。

### レイアウト(iPad横向き1180×820pt前提)

- 左〜中央: 児童グリッド(7列基本、38人まで1画面・スクロールなし)
  - セル: min-height 46px以上、出席番号(小)+姓+状態記号
  - 状態済みセル: 該当する状態色(bg/fg/bd)
  - 入力中の児童: `--st-cur-bd` 2px枠
- 右側固定: 状態パレット(幅150〜170px)
  - 「入力中: ◯番 ◯◯」表示(--fs-title)
  - 状態ボタン: 提出(緑)/遅れ(橙)/未提出(赤)を min-height 52px、欠席を44px
  - 進捗カウンタ「24 / 35」常時表示
- 既存の状態種別(現アプリの提出物ステータス定義)に合わせること。上記3+1種と異なる場合は現行定義を優先し色を割り当てる

### 動作

1. 画面表示時、最初の未入力児童が「入力中」になる
2. 状態ボタンタップ → 現在の児童に記録(既存の保存関数を呼ぶ) → **次の未入力児童へ自動前進**(出席番号順、末尾まで行ったら先頭から未入力を探す)
3. グリッド上の任意の児童をタップ → その児童へジャンプ(割り込み修正。入力済み児童も選択でき、上書き可能)
4. 全員入力完了 → トースト「全員入力完了 ◯/◯」+ 入力中ハイライトを消す
5. 自動前進のON/OFFトグルは**今回は実装しない**(試用後に必要なら追加)

### 既存UIとの関係

- 既存のチェックUI(一覧形式等)が別の用途で使われている場合は壊さず、連続入力モードを既定表示とし、既存表示への切替手段を残すかは既存構造を見て判断・報告すること
- 色だけに頼らず記号(✓/遅/未/欠 ※現行の記号定義があればそれに従う)を必ず併記

### 検証

1. 38人クラスのモックデータで1画面に収まること(スクロールが発生しないこと)
2. 連続入力→全員完了→トーストの一連フロー
3. 入力済み児童をタップ→状態を変更→正しく上書き保存されること
4. 保存データの形式が変更前と同一であること(既存の集計・成績連携が壊れないこと)
5. node --check / 実ブラウザでコンソールエラーなし

### 注意

- データ保存形式は一切変更しない(grdGetSubmissionScore10等の成績連携が読む形式を維持)
- 試用期間のため、派手なアニメーションは入れない(タップへの応答は即時の色変化のみ)
